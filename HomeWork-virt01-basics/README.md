# Задача 2. Выбор типа платформы виртуализации
Критерии выбора: Отказоустойчивость, производительность (особенно дисковая и процессорная), совместимость с оборудованием и ОС, простота управления и соответствие законодательству.

Задача	Рекомендуемый тип платформы	Объяснение критериев выбора
Высоконагруженная БД MySQL, критичная к отказу	Паравиртуализация (VMware vSphere, Hyper-V, KVM)	Критерии: Производительность и отказоустойчивость.
«Физические сервера» часто используют для БД, но они проигрывают в гибкости восстановления при отказе. Современная паравиртуализация (KVM, ESXi) дает производительность, близкую к нативной (особенно с использованием PV-драйверов). Кластер виртуализации (vSphere HA/FT) решит проблему критичности к отказам быстрее, чем поднятие БД с бэкапа на физическом сервере.
Различные web-приложения	Виртуализация уровня ОС (Docker, Containerd) + Паравиртуализация	Ключевое слово: «различные». Для микросервисов или приложений на разных стеках (PHP, Python, Node.js) контейнеры идеальны — они легковесны и быстро масштабируются. Если приложения монолитные или требуют специфичных ядер ОС, лучше использовать традиционные ВМ на базе паравиртуализации для изоляции.
Windows-системы для бухгалтерии	Паравиртуализация (Hyper-V или VMware)	Критерии: Совместимость, простота управления, лицензирование.
Windows требует своей ОС и ядра. Контейнеры Windows (на базе виртуализации ОС) существуют, но они сложнее в настройке и менее распространены. Паравиртуализация гарантирует полную совместимость с любым бухгалтерским ПО (1С, Контур и т.д.) и упрощает создание снэпшотов перед обновлениями.
Системы с высокопроизводительными расчётами на GPU	Паравиртуализация с поддержкой PCI Passthrough	Критерии: Прямой доступ к железу.
Виртуализация уровня ОС (контейнеры) имеет ограниченный и более сложный доступ к GPU (хотя это решаемо через NVIDIA Container Toolkit). Паравиртулязация с технологией PCI Passthrough (проброс видеокарты) или vGPU дает виртуальной машине прямой доступ к вычислительным мощностям GPU, что критически важно для производительности расчетов (рендеринг, AI/ML).
Задача 3. Выбор системы управления виртуализацией
Сценарий	Рекомендуемое решение	Обоснование
100 ВМ (Linux/Windows), преимущественно Windows, нужны балансировщики, репликация, бекапы.	Microsoft Hyper-V (в составе Windows Server) или VMware vSphere	Почему Hyper-V: По условию, инфраструктура преимущественно Windows-based. Hyper-V нативно интегрируется с Active Directory, PowerShell (автоматизация), а технологии типа Storage Replica и Failover Clustering идут «из коробки» или встроены в Windows Server. Балансировщики можно поднять как на сторонних решениях, так и на NLB от Microsoft. Для бекапов отлично подойдет Veeam, который заточен под Hyper-V/VMware. VMware дал бы те же возможности, но был бы сложнее в лицензировании при таком количестве Windows-гостевых ОС (лицензии Windows оплачиваются отдельно в любом случае).
Бесплатное open source решение для 20-30 серверов (Linux + Windows).	Proxmox VE	Proxmox базируется на KVM (паравиртуализация) и LXC (контейнеры). Это полностью open source, имеет удобный веб-интерфейс, встроенные механизмы бекапа и поддержку кластеризации. Отлично работает и с Windows, и с Linux. Бесплатное решение, которое тянет функционал коммерческих продуктов начального уровня.
Бесплатное, совместимое и производительное решение для Windows-инфраструктуры.	Microsoft Hyper-V Server	Это бесплатная редакция Hyper-V (отдельный продукт, без графической оболочки Windows). Она поддерживает те же гостевые Windows, что и платный Windows Server, обеспечивая максимальную совместимость (драйверы Integration Services идут сразу). Управление осуществляется удаленно через Windows Admin Center или RSAT. KVM отлично работает с Windows, но требует больше «танцев с бубном» для установки драйверов и настройки производительности. Hyper-V Server проще в развертывании для чисто Windows-среды.
Рабочее окружение для тестирования ПО на нескольких дистрибутивах Linux.	Виртуализация уровня ОС (Docker) + Vagrant или Oracle VirtualBox	Вариант 1 (профессиональный): Если тестируются сетевые взаимодействия или микросервисы, лучше Docker. Вариант 2 (универсальный): VirtualBox (паравиртуализация для десктопа) + Vagrant. Vagrant позволяет поднять несколько разных дистрибутивов (Ubuntu, CentOS, Fedora) одной командой, идеально подходит для разработчика/тестировщика на локальной машине.
Задача 4. Гетерогенная среда виртуализации
Возможные проблемы и недостатки:

Сложность управления (Admins Complexity): Администраторам нужно знать два разных стека технологий (например, команды PowerShell для Hyper-V и команды esxcli/vCenter для VMware). Разные консоли управления.

Разрозненные инструменты бекапа и мониторинга: Придется использовать разные агенты или разные настройки в Veeam/Zabbix, либо искать универсальное ПО, которое дороже.

Проблемы с сетью (SDN): Сложнее настроить единую политику безопасности или виртуальные сети между платформами. Потребуется "сшивать" их на физическом уровне или шлюзами.

Лицензирование (Vendor Lock-in & Cost): Усложняется учет лицензий (особенно Microsoft в гостевых ОС при миграции между разными гипервизорами).

Миграция (V Motion): Невозможна "горячая" миграция между разными платформами. Для перемещения ВМ с VMware на Hyper-V нужен конвертер и простой.

Минимизация рисков:

Автоматизация и оркестрация: Внедрение слоя управления поверх гипервизоров (например, OpenStack, Kubernetes для контейнеров, Terraform для поставки инфраструктуры). Это скроет различия от инженеров эксплуатации.

Централизованный мониторинг: Использование систем, умеющих собирать метрики с разных гипервизоров по API (Zabbix, Prometheus + Exporters).

Унификация хранения: Использование общих протоколов (NFS, iSCSI) для обоих гипервизоров, чтобы данные были доступны, но с пониманием блокировок.

Четкое зонирование ответственности: Разделить кластеры по назначению (например, VMware только под критичные БД, Hyper-V только под RDS-фермы), чтобы не плодить хаос в одном месте.

Если бы у вас был выбор, создавали бы вы гетерогенную среду или нет?

Нет. Если есть возможность выбора и нет жестких внешних причин (например, поглощение компании, специфическое ПО, работающее только на определенном гипервизоре), лучше создать гомогенную среду.
Это дешевле в эксплуатации, проще в обучении персонала и значительно надежнее с точки зрения предсказуемости отказов. Гетерогенность оправдана только в двух случаях:

Политика вендоров: Сознательное избегание привязки к одному вендору (стратегия "multi-vendor").

Техническая необходимость: Например, один гипервизор лучше работает с GPU (KVM), а второй идеально держит легаси Windows-приложения (Hyper-V).
